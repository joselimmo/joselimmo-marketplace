## Background

Story 2.5 closed the CLI's human-readable surface: walker, formatter, exit codes, integration tests. The remaining gap before plugin authors and CI integrators can adopt Caspian programmatically is the machine-readable output (`--format=json`) and the publish-artifact integrity gate (`verify-pack`). FR8, NFR11, NFR15, NFR19, NFR20 hinge on these two deliverables.

The B4 schema is the **first user-facing contract** of the validator. Once published, downstream consumers (GitHub Actions workflows, jq pipelines, third-party dashboards) will hard-code `.summary.errors` / `.results[].diagnostics[].code` lookups. A breaking change to this shape post-v1.0 is a coordination tax measured in months. Story 2.6 therefore over-invests in shape stability: explicit `schemaVersion: "1"`, golden snapshot tests that fail on any shape drift, and a documented deprecation policy.

The `verify-pack` snapshot is the **first publish-artifact integrity gate**. Today, `pnpm pack --dry-run` from `packages/cli/` emits 28 files (see CC8 below for the current list, including `dist/.tsbuildinfo` and 9 source-map / `.d.ts.map` pairs). Without a snapshot, an inadvertent change to `package.json#files` or a future TypeScript build flag could silently bloat the published tarball or — worse — leak source paths or test fixtures. The `verify-pack` script locks the current set as the baseline. Anything added or removed fails CI until the snapshot is intentionally updated.

### AC3 — Invalid `--format=<value>` exits 2 with a usage error

**Given** an invalid format value

**When** I run `node packages/cli/dist/cli.js validate --format=xml ./fixtures/valid/`

**Then** the CLI exits `2`

**And** stderr contains exactly: `error: invalid --format value: 'xml' (expected 'human' or 'json')` followed by `Run 'caspian validate --help' for usage.`

**And** stdout is empty (no partial JSON, no human output) — usage errors route to stderr per architecture.md:432.

**Detection rule:** the validation happens at the top of `runValidate(input, options)` BEFORE the walker runs. Invalid formats are detected as `options.format !== "human" && options.format !== "json"`. The handler returns `EXIT_USAGE` after writing the message to stderr.

### AC4 — `--format=json` produces a top-level B4-shaped object

**Given** the `--format=json` flag

**When** I run `node packages/cli/dist/cli.js validate --format=json ./fixtures/valid/core-overview/minimal.md`

**Then** stdout is **valid JSON** (parseable by `JSON.parse(stdout)` without throwing)

**And** the parsed object has exactly three top-level keys, in this order: `schemaVersion`, `results`, `summary`

**And** `parsed.schemaVersion === "1"` (string, not number — easier forward-compat with `"1.1"` migration markers)

**And** `parsed.results` is an array of `Result` objects (one per validated file)

**And** `parsed.summary` is an object with three fields: `files`, `errors`, `warnings` (all numbers)

**And** the human formatter is **bypassed entirely** — no per-file blocks, no `(no diagnostics)` lines, no summary footer text, no ANSI escapes. The only thing on stdout is the JSON document plus exactly one trailing `\n`.

### AC5 — Each `Result` has the B4-mandated shape

**Given** a `Result` entry in `parsed.results`

**When** I inspect any element

**Then** the keys are present in this insertion order: `file`, `valid`, `diagnostics`

**And** `result.file` is a string — the **display path**, normalized identically to the human formatter:
- Relative path with forward slashes when the file lives under `cwd` (e.g., `fixtures/valid/core-overview/minimal.md`).
- Absolute path with forward slashes when the file is outside `cwd`.
- `.` when the file is `cwd` itself (defensive — should never happen in normal flow).

**And** `result.valid` is a boolean: `true` iff `result.diagnostics` contains zero entries with `severity === "error"`. Files with only warnings are `valid: true`. **Important:** `valid` mirrors the warnings-allowed exit-code-0 contract from Story 2.5 AC4 — warnings do NOT make a file "invalid". Use `severity` filtering, not `length === 0`.

**And** `result.diagnostics` is an array (possibly empty) of `Diagnostic` objects in the order returned by `validateFile()` (the core's pipeline order; do NOT sort).

### AC6 — Each `Diagnostic` has the B4-mandated shape

**Given** a `Diagnostic` entry in `result.diagnostics`

**When** I inspect any element

**Then** the keys are present in this insertion order: `code`, `severity`, `line`, `field?`, `message`, `doc?`

**And** `diagnostic.code` is a string matching `/^CASPIAN-[EW]\d{3}$/` (e.g., `"CASPIAN-E008"`, `"CASPIAN-W001"`).

**And** `diagnostic.severity` is exactly `"error"` or `"warning"` (no `"info"`, no `"hint"`).

**And** `diagnostic.line` is a positive integer (matches `Diagnostic.line` from `@caspian-dev/core`).

**And** `diagnostic.field` is **present iff** `Diagnostic.field` from core is a non-empty string. If the core diagnostic has no `field` (e.g., E001 BOM, E002 encoding, E004 oversized), the `field` key is **omitted** from the JSON object — not emitted as `null` or `""`. Implementation: `const { field, ...rest } = src; const out = field ? { ...rest, field } : rest;` after building the rest of the object in the correct order. **Or** explicit conditional spread: `{ code, severity, line, ...(field ? { field } : {}), message, ...(doc ? { doc } : {}) }`.

**And** `diagnostic.message` is the **verbatim** message from `Diagnostic.message` returned by core. **No** hint-extraction, **no** stripping of the `". Did you mean \`metadata\`?"` suffix that W001 emits. The JSON formatter preserves the canonical message text byte-for-byte; downstream consumers can parse the suggestion themselves if needed. (Rationale: stripping in JSON would couple the contract to Story 2.4's W001 message format. Adding a `hint` field is a non-breaking schema "1" enhancement reserved for future work.)

**And** `diagnostic.doc` is **present iff** the code is a known `CASPIAN_*` constant (i.e., `getDocUrl(code) !== undefined`). For all 18 v1.0 codes (E001–E014 + W001–W004), `doc` is always present, sourced from `DIAGNOSTIC_DEFINITIONS[code].doc` (e.g., `"https://caspian.dev/diagnostics#caspian-e008"`). Unknown codes (which should never happen in v1.0) result in `doc` being **omitted** — not `null`, not `""`, not present.

### AC7 — `summary` aggregates totals correctly

**Given** the `summary` object

**When** I inspect it

**Then** `summary.files === parsed.results.length` (exact equality; counts walked-and-validated files only — does NOT include `walker.skippedOutsideCwd` or `walker.skippedUnresolvable`).

**And** `summary.errors === sum(r.diagnostics.filter(d => d.severity === "error").length for r in results)` (total error diagnostics across all files).

**And** `summary.warnings === sum(r.diagnostics.filter(d => d.severity === "warning").length for r in results)` (total warning diagnostics across all files).

**And** the three counts are non-negative integers; on a fixtures-empty-glob path, `runValidate` exits 2 BEFORE the formatter runs, so `summary.files === 0` is unreachable in v0.1 via normal CLI flow (but the formatter contract is `{ files: 0, errors: 0, warnings: 0 }` if it ever does).

**And** `summary` does **NOT** include a `skipped` field in v1 schema. (Skipped-file warnings still emit on stderr in JSON mode — same behavior as human mode. `summary.skipped` is reserved as a non-breaking additive field for a future schema "1" minor enhancement.)

### AC19 — `scripts/verify-pack.ts` is the CI gate

**Given** the `verify-pack` script

**When** I open `caspian/packages/cli/scripts/verify-pack.ts`

**Then** the script:

1. Spawns `pnpm pack --dry-run --json` from the CLI package directory using `child_process.execFileSync("pnpm", ["pack", "--dry-run", "--json"], { cwd: ..., shell: true })` (`shell: true` is required on Windows for `.cmd` shim invocation, matching the Story 2.5 P7 pattern at `cli-end-to-end.test.ts:18`).
2. Parses the stdout as JSON.
3. Extracts `result.files` and reduces each entry to `{ path: string }` — discarding any other fields the pnpm output may contain (e.g., `mode`, `size` — neither is currently emitted by pnpm 10 but the reducer is forward-compatible).
4. Sorts the extracted array by `path` (codepoint order).
5. Loads `tests/integration/published-files.snapshot.json`, parses, sorts the `files` array identically.
6. Compares the two sorted arrays. **Match criterion:** identical length AND every entry's `path` equals at the same index.
7. **On match:** prints `verify-pack: OK (N files match snapshot)` and exits `0`.
8. **On drift:** prints a unified diff (added paths prefixed `+`, removed paths prefixed `-`) to stderr along with the message:
   ```
   verify-pack: FAIL — published file list drift detected.
   To update the snapshot intentionally, run: pnpm -F @caspian-dev/cli pack --dry-run --json
   then copy the sorted `files` array into tests/integration/published-files.snapshot.json
   ```
   Then exits `1`.
9. **On any other error** (pnpm not on PATH, snapshot missing, snapshot malformed): prints the error to stderr and exits `1` (treat as drift — defensive default).

**Implementation skeleton** (reference; adapt to project style):

```ts
import { execFileSync } from "node:child_process";
import { readFileSync } from "node:fs";
import path from "node:path";
import { fileURLToPath } from "node:url";

const here = path.dirname(fileURLToPath(import.meta.url));
const PACKAGE_ROOT = path.resolve(here, "..");
const SNAPSHOT_PATH = path.resolve(
  PACKAGE_ROOT,
  "tests",
  "integration",
  "published-files.snapshot.json",
);

interface PackResult {
  files: Array<{ path: string }>;
}

interface Snapshot {
  files: Array<{ path: string }>;
}

function sortByPath<T extends { path: string }>(arr: T[]): T[] {
  return [...arr].sort((a, b) => (a.path < b.path ? -1 : a.path > b.path ? 1 : 0));
}

function diff(actual: string[], expected: string[]): string[] {
  const lines: string[] = [];
  const a = new Set(actual);
  const b = new Set(expected);
  for (const p of expected) if (!a.has(p)) lines.push(`- ${p}`);
  for (const p of actual) if (!b.has(p)) lines.push(`+ ${p}`);
  return lines;
}

const raw = execFileSync("pnpm", ["pack", "--dry-run", "--json"], {
  cwd: PACKAGE_ROOT,
  encoding: "utf-8",
  shell: true,
}).trim();

const pack = JSON.parse(raw) as PackResult;
const snapshot = JSON.parse(readFileSync(SNAPSHOT_PATH, "utf-8")) as Snapshot;

const actualPaths = sortByPath(pack.files.map((f) => ({ path: f.path }))).map((f) => f.path);
const expectedPaths = sortByPath(snapshot.files).map((f) => f.path);

const driftLines = diff(actualPaths, expectedPaths);
if (driftLines.length === 0) {
  console.log(`verify-pack: OK (${actualPaths.length} files match snapshot)`);
  process.exit(0);
}

console.error("verify-pack: FAIL — published file list drift detected.");
for (const line of driftLines) console.error(line);
console.error(
  "To update the snapshot intentionally, run: pnpm -F @caspian-dev/cli pack --dry-run --json",
);
console.error(
  "then copy the sorted `files` array into tests/integration/published-files.snapshot.json",
);
process.exit(1);
```

**Test exemption:** `verify-pack.ts` is a script, not a vitest test. It is invoked via `pnpm -F @caspian-dev/cli verify-pack` (and the root-level convenience script `pnpm verify-pack`). It is intentionally NOT covered by a vitest test — the script's correctness is verified by running the gate itself in CI.

**Size delta deferral:** the epic AC mentions "size delta beyond a small tolerance". `pnpm pack --dry-run --json` (pnpm 10) does NOT emit per-file or total-tarball sizes in its output. Adding a separate `pnpm pack` (without `--dry-run`) into a temp dir + `fs.statSync().size` check would significantly slow the CI gate and require careful temp-file cleanup. **Decision: defer total-tarball size tracking to a future hardening story.** v0.1 verify-pack diffs the file list only. Document this scope reduction in *Completion Notes*. (If `pnpm pack --json` adds size fields in a future minor, a follow-up story can extend the snapshot schema to `{ "files": [{ "path": "...", "size": N }, ...] }` non-breakingly.)

### Schema (v1)

```json
{

- `schemaVersion` is the stable contract identifier. v1.0 ships `"1"`.
- `results[].file` is the path as walked, with forward slashes regardless of OS.
- `results[].valid` is `true` iff no `severity: "error"` diagnostic is present (warnings are allowed).
- `diagnostics[].field` is omitted when the diagnostic does not target a specific field (e.g., `CASPIAN-E001` BOM).
- `diagnostics[].doc` is omitted only for unknown codes (not present in the v1.0 18-code registry — should never occur).
- `summary.files` counts walked-and-validated files (not skipped files; skipped files emit warnings on stderr).

### Schema stability

`schemaVersion: "1"` is a stable contract. Adding optional fields (e.g., a future `summary.skipped` count or `diagnostic.hint` field) is **non-breaking** and does NOT bump the version. Removing or renaming any field, changing a field's type, or changing the semantics of an existing field bumps `schemaVersion` to `"2"` (or higher).

When `schemaVersion` is bumped, the prior version is documented as deprecated in `CHANGELOG.md` for at least one minor release before the old shape is removed. Downstream consumers should pin against `parsed.schemaVersion === "1"` and emit a warning when they encounter a different value.

### Strict-warnings recipe

Default exit semantics: `caspian` exits 0 when only warnings are present. To gate CI on a zero-warnings, zero-errors invariant, pipe through `jq`:

```bash
caspian validate --format=json ./skills/ | jq -e '.summary.errors == 0 and .summary.warnings == 0'
```

`jq -e` exits non-zero when the predicate evaluates to `false`, propagating a failing exit code to the surrounding CI step.

### Implementation Decision — `output/types.ts` vs inline `FileResult`

Two approaches:

**Option A:** Keep `FileResult` inline in both `human.ts` and `json.ts`, declared independently, kept consistent by code review.
- **Pros:** Zero new files; minimal change.
- **Cons:** Type drift risk — adding a field to one declaration silently breaks the other's type compatibility at the call site (since `FileResult` is structural). A future story adding `result.summary` or `result.skipped` to one but not the other would compile-pass but introduce subtle bugs.

**Option B:** Extract `FileResult` to `src/output/types.ts`; both formatters import it.
- **Pros:** Single source of truth; type drift impossible by construction.
- **Cons:** Adds one trivial 5-line file.

**Decision: Option B.** The cost (one new file) is trivial; the protection (compile-time enforcement of formatter input compatibility) is durable. Future formatters (e.g., `--format=junit-xml` in a later story) inherit the same input contract for free.

### Implementation Decision — `verify-pack` size tracking

Two approaches considered:

**Option A:** Track only the file list (`{ path: string }[]`). Drift on adds/removes only.
- **Pros:** Zero CI overhead beyond `pnpm pack --dry-run --json`. Snapshot is human-readable + diff-friendly. Forward-compatible with adding `size` later.
- **Cons:** A silent file-content bloat (e.g., a TypeScript build flag emitting larger source maps without changing the file list) is not detected.

**Option B:** Track total tarball size with tolerance.
- **Pros:** Catches content bloat.
- **Cons:** Requires running real `pnpm pack` (not `--dry-run`) into a temp directory. Adds CI overhead. Requires temp-file cleanup. Tolerance is hand-tuned and may need adjustment as the package grows.

**Decision: Option A for v0.1.** Story 2.6's primary goal is "the file list never silently changes," which Option A satisfies. Content-bloat detection is a hardening improvement; defer to a future story (or to the existing `pnpm pack --dry-run --json` ever gaining a `size` field, which would let Option A non-breakingly extend its schema). Document the deferral in *Completion Notes*.

### Implementation Decision — JSON pretty-print vs minified

**Decision: pretty-print with 2-space indent.**

Rationale: golden snapshots compared via `expect(stdout).toMatchSnapshot()` are read by humans during snapshot review (the `git diff` of `format-json.test.ts.snap`). A minified single-line JSON document on a 19-fixture run produces a ~400-character single line — unreadable in diff review and hard to manually sanity-check. The 2-space pretty-print costs ~30% extra bytes (negligible — single-digit kilobytes for typical CI runs) and pays for itself in review ergonomics.

### Implementation Decision — `field` and `doc` omission vs explicit `null`

**Decision: omit absent optional fields entirely.**

Rationale:
- TypeScript optional fields (`field?: string`) idiomatically map to "key absent" in JSON, not `"field": null`.
- jq pipelines like `jq '.results[].diagnostics[] | select(.field == "metadata")'` work cleanly when the key is absent (the comparison evaluates to `false` deterministically); a `null` value would require `select(.field // "" == "metadata")` defensiveness.
- The TypeScript compile-time `Diagnostic.field?: string` shape is preserved; consumers reading the JSON via TypeScript get the same `.field?: string` ergonomics.
- Adding `null` later would be a non-breaking schema "1" enhancement (consumers ignoring null are still correct); going from `null` to omission later is a breaking change. Conservative direction wins.

### Walker order vs JSON results array order

The JSON `results[]` array order MUST match the walker's discovery order — same as the human formatter's per-file block order. Story 2.5 AC2 documents this as "filesystem-order on Linux, alphabetical on macOS/Windows; do NOT sort." The JSON formatter trusts the input array order and never sorts. **Do NOT** alphabetize results inside `formatJson`; that would make stdout differ from `formatHuman`'s order and break the dual-output-channel-equivalence invariant.

### `pnpm pack --dry-run --json` quirks

- Output is a single JSON object with `name`, `version`, `filename`, `files[].path`. **No** `size` per file in pnpm 10. **No** stable sort of `files[]` (verifier sorts before comparing).
- `filename` contains the build machine's absolute path (e.g., `F:\\work\\joselimmo-marketplace-bmad\\caspian\\caspian-dev-cli-0.0.1.tgz`). The verifier MUST ignore `filename`.
- On Windows, invoking `pnpm` via `execFileSync` requires `shell: true` (matches the Story 2.5 P7 pattern at `tests/integration/cli-end-to-end.test.ts:18`). On Linux/macOS, `shell: true` is harmless but not strictly required.
- The `--dry-run` flag means no `.tgz` is written to disk. If `shell: true` somehow strips `--dry-run` on a future pnpm version, a stray `caspian-dev-cli-*.tgz` could land in the working tree — Task 9's `.gitignore` entry guards against accidental commits.
