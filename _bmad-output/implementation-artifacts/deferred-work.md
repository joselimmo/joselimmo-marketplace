## Deferred from: code review of 1-4-envelope-json-schema-draft-2020-12 (2026-04-27)

- **`count` field name implies exact count; description says "minimum"** ‚Äî Field is named `count` in `core.md`; runtime semantics are "at-least-N". Renaming (e.g., to `min_count`) is a breaking spec change and out of scope for Story 1.4. Revisit at spec v0.2+ RFC. [`caspian/schemas/v1/envelope.schema.json` ‚Äî `$defs.RequiresEntry.count`]
- **`schema_version` accepts empty string (no `minLength`)** ‚Äî Schema-level rejection of unrecognized versions would convert CASPIAN-W003 (warn-on-unknown) into an error, breaking the graceful-degradation contract (NFR16). Empty/unrecognized values handled by the warning system in Story 2.4. No change needed unless the warn-on-unknown policy changes. [`caspian/schemas/v1/envelope.schema.json` ‚Äî `properties.schema_version`]
- **`tags` array lacks `minItems: 1`, `uniqueItems: true`, `items.minLength: 1`** ‚Äî Empty tags array is semantically equivalent to omitting the field; duplicate/empty-string tags are noise. Not addressed by any Story 1.4 AC. Could be tightened in a future schema hardening story (post-v1.0 or at Story 1.6 fixture authoring if needed). [`caspian/schemas/v1/envelope.schema.json` ‚Äî `$defs.RequiresEntry.tags`]
- **`requires: []` (empty array) passes validation** ‚Äî `minItems: 1` not required by any AC. An empty requires array is semantically equivalent to omitting the field; open design question whether it should be forbidden at schema level. [`caspian/schemas/v1/envelope.schema.json` ‚Äî `properties.requires`]
- **`caspian/LICENSE` leading blank line (Story 1.1 artifact)** ‚Äî Both `caspian/LICENSE` and `caspian/schemas/LICENSE` open with a blank line before the Apache License header. Byte-identical per `diff`. Non-standard vs. canonical Apache 2.0 text; may cause SPDX tooling mismatch. Cleanup belongs in a housekeeping commit or Story 1.1 follow-up. [`caspian/LICENSE`, `caspian/schemas/LICENSE`]
- **`schema_version: 0.1` YAML float-cast trap** ‚Äî Unquoted `schema_version: 0.1` in YAML frontmatter is parsed as float `0.1` by YAML 1.1 auto-cast, causing a type error against the `"type": "string"` schema constraint. Fix is a documentation/prose note in the spec (`core.md`) reminding authors to quote the value. Handled by the YAML pipeline in Story 2.3. [`caspian/spec/core.md` ‚Äî `schema_version` authoring guidance]
- **No upper bound on `count` integer** ‚Äî `count` has `minimum: 1` but no `maximum`. Very large integers pass. No practical upper bound required by any AC or architecture constraint; could add `maximum` in a future hardening story if tooling needs it. [`caspian/schemas/v1/envelope.schema.json` ‚Äî `$defs.RequiresEntry.count`]
- **`Produces.additionalProperties: false` limits future extensibility** ‚Äî Strict `additionalProperties: false` means adding any new field to `produces` (e.g., `schema_version`, `tags`) requires a schema major bump per NFR22 BACKWARD_TRANSITIVE policy. This is the intentional Design A5 asymmetry (inner shapes locked, outer envelope open). Future spec evolution must introduce a `v2/` directory. Not actionable in Story 1.4. [`caspian/schemas/v1/envelope.schema.json` ‚Äî `$defs.Produces`]
