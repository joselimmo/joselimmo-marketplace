## 3. What Went Well

### 3.1 Reference Models pattern matured organically

Stories 1.1 and 1.2 needed 3 and 14 patches respectively after BMad code review. Stories 1.5, 1.6, 1.7 needed 0, 2, 0 patches. The differentiator was the **Reference Models** subsection in Dev Notes — byte-faithful starting points for every file the dev would create. By Story 1.6, 25 Reference Models drove the implementation with only one deviation (the `vendor-namespaced.md` AC6 premise reversal, which was caught by code review and is now part of the body of knowledge). By Story 1.7, every file had a Reference Model and the only deviation was the `LICENSE.md` optional sub-task (deliberately scoped as optional in the story).

### 3.2 Smoke gate baseline preserved across the epic

The 7-file biome lint baseline established at Story 1.5 was preserved exactly through Stories 1.6 and 1.7 with no `caspian/biome.json` or `caspian/.biomeignore` modification. The architecture decision to keep `**/*.md` out of `files.includes` (and to negate `fixtures/invalid/**`) was prescient — markdown-heavy stories did not pollute the lint scope, and JSON expectation files in invalid fixtures stayed out of biome's reach.

### 3.3 Spec contract sealed cleanly via review-driven decisions

Story 1.2's 3 decisions resolved into patches D1 (split-on-first-colon `type` parsing), D2 (`requires.count` = at-least-N, validator-non-enforced), and D3 (`core:<undocumented>` = warning) sealed `caspian/spec/core.md` with no subsequent retraction. The contract is stable for Epic 2's validator implementation.

### 3.4 Forward-reference annotation discipline worked

The convention of writing `*(coming soon — Story X.Y)*` next to links to not-yet-existent files allowed each story to reference its future siblings without breaking the GitHub UX. Sealed files (Story 1.2's `spec/core.md`, Story 1.3's vocabulary docs) were never edited retroactively — the annotations turned into historical project state markers, and the links resolved naturally as target stories merged.

### 3.5 Per-directory LICENSE re-declaration (CNCF/Kubernetes pattern)

Stories 1.4, 1.5, 1.6 each added an Apache-2.0 byte-faithful copy at `caspian/{schemas,diagnostics,fixtures}/LICENSE`. Story 1.7 deliberately deviated with a dual-statement `caspian/examples/LICENSE.md` (CC-BY-4.0 prose + Apache-2.0 SKILL.md artifacts) because `examples/` is mixed-license per architecture line 608. The deviation was correct; the discipline of "re-declare LICENSE explicitly when a directory may be consumed in isolation" held across the whole epic.

### 3.6 Defensive YAML quoting discipline internalized

Discovered on Story 1.4 (the `schema_version: 0.1` float-cast trap, deferred). Reappeared on Story 1.6 (the `E007-unquoted-bool/` invalid fixture exhibits the YAML 1.1 boolean coercion). Story 1.7 defensively quotes all version-like and boolean-like values (`"0.1"`, `"1.0"`, `"1.0.0"`, `"true"`, `"false"`). Three stories to internalize the reflex; now part of the architecture's *YAML Frontmatter Authoring* convention (lines 402–451).

## 4. Challenges and Growth Areas

### 4.1 Drift between PRD / architecture / epics canonical paths

Two material discrepancies surfaced:

- PRD line 556 says `spec/examples/minimal-skill-adoption/`; architecture line 610 + epics line 668 say `examples/minimal-skill-adoption/` (top-level). Story 1.7 resolved to top-level `examples/` per the more recent architecture + epics.
- Architecture line 391 says `all-22-fields.md`; epics line 636 says `all-22-known-fields.md`. Story 1.6 followed epics.

The team's heuristic that emerged: **epics > architecture > PRD** when there is divergence (because epics is the most recently authored canonical AC source, and architecture supersedes PRD). This rule is now applied story-by-story but is not codified anywhere. Action item: document this precedence in `CLAUDE.md` or a similar repo-level doc.

### 4.2 Stale forward-references in sealed files

Story 1.2 wrote 5 `*(coming soon — Story 1.3)*` annotations on links to `vocabulary/`. Story 1.3 delivered, but `spec/core.md` was sealed and not edited. The annotations became stale. Same pattern on Story 1.6 — 7 vocabulary docs (`adr.md`, `convention.md`, `learning.md`, `glossary.md`, `review.md`, `rule.md`, `scratch.md`) reference `core-X/` fixtures that were **out of scope for Story 1.6** and are not planned for any other Epic 1 story.

The forward-reference discipline serves us when the target story is in-flight; it produces noise when the target is deferred indefinitely. Action item: a "forward-reference cleanup" sweep at the end of each epic.

### 4.3 The `biome.json` strict-JSON trap

Story 1.1 review attempted to add a `// TODO Story 2.1: enforce schemas lockdown via biome` breadcrumb comment in `biome.json`. Biome 2.4.13 parses `biome.json` as strict JSON (not JSONC despite the file's permissive feel). The comment caused `pnpm lint` to fail with cascading parse errors. Reverted. The breadcrumb was placed in three text-searchable locations instead (`deferred-work.md` + Review Findings + Dev Agent Record).

The lesson: **`deferred-work.md` is a first-class artifact, not just a tracking log**. A future contributor grepping for "Story 2.1" finds the breadcrumb across multiple stories. Action item: keep deferred-work.md as the canonical breadcrumb mechanism for cross-story future work.

### 4.4 No runtime tests yet — validator hypotheses are unverified

The 17 `caspian/fixtures/invalid/<code>/<variant>.expected.json` files declare expected diagnostics with line numbers. Nobody has executed them. Story 2.3/2.4 will load them, and the team will discover whether conventions like `E002 line:1` (file-level rejection), `E004 line:1` (oversized frontmatter), `E008 line:1` (absent-field anchor) hold. These line-number conventions are documented in deferred-work.md as known unknowns to be resolved by Story 2.6 golden snapshots.

### 4.5 The `deferred-work.md` file is growing fast

36 entries already across 5 reviewed stories. Many target Epic 2 stories explicitly (2.1, 2.2, 2.3, 2.4, 2.6, 2.8). Some target spec evolution (`v0.2+` RFCs). Without periodic triage, individual entries can become forgotten landmines. Action item: triage `deferred-work.md` at every epic boundary.

## 10. Key Takeaways

1. **The Reference Models pattern is the highest-leverage quality lever.** Patch count dropped from 14 (Story 1.2, no Reference Models) to 0 (Story 1.7, every file modeled). Continue the pattern in Epic 2 even when the deliverable is code instead of content.
2. **`deferred-work.md` works as a first-class artifact when treated as a breadcrumb mechanism.** It caught all three critical-path discoveries that surfaced during this retrospective. Continue triaging it at every epic boundary.
3. **Forward-reference discipline serves us when the target story is in-flight; it produces stale annotations when the target is deferred or out of scope.** Add a forward-reference cleanup sweep to each epic's exit checklist.
4. **The spec / architecture / PRD divergence is real and recurring.** Codify the **epics > architecture > PRD** precedence rule in repo-level conventions.
5. **Defensive YAML quoting (versions, booleans, anything float-castable or 1.1-coercible) is now muscle memory.** Carry forward into casper-core skill / command frontmatter authoring (Epic 3).
6. **Branding-level dependencies (npm name, domain, organization handles) deserve a dedicated pre-flight at every epic boundary.** Discovery 1 (npm `caspian` squatted) would have blocked Story 2.8 mid-flight if not surfaced now.
